# 会话描述统一方案

> Type: `inprogress`
> Updated: `2026-04-14`
> Summary: 统一 Thread 用户可见描述，默认去除短 ID 暴露，并定义分场景展示策略与分阶段实施路径。

## 背景

当前同一个会话在不同入口使用了不同描述来源（标题、preview、短 ID、状态文案），导致用户在 `/use`、切换提示、确认卡、`/status` 之间看到的“会话身份”不一致。

## 目标

1. 默认不向普通用户展示短 ID。
2. 统一会话描述为四类语义信息：
   - 标题
   - 会话起点（首条用户消息）
   - 最近用户
   - 最近回复
3. 按场景选择信息密度，低风险场景简洁，高风险/调试场景信息更全。

## 用户可见结果（最终态）

### `/use` 会话选择

- 主信息：标题
- 辅助信息：最近用户（没有则退回会话起点，再退回最近回复）
- 状态信息：可接管/已接管/占用 + 相对时间
- 默认不显示短 ID

### 会话切换成功提示卡

- 当前输入目标：标题
- 会话起点
- 最近用户
- 最近回复
- 默认不显示短 ID

### 强踢确认卡

- 标题
- 会话起点
- 最近用户
- 最近回复
- 不显示短 ID

### `/status`

- 当前会话：标题
- 会话起点
- 最近用户
- 最近回复
- 最近活跃（相对时间）
- 默认不显示短 ID（仅诊断模式或标题冲突时可选显示）

### 请求确认卡（request prompt）

- 保持简洁：当前会话标题 + 请求正文
- 默认不显示短 ID

## 分阶段实施

### 阶段 1：摘要数据打底

- 在会话记录层补齐“会话起点 / 最近用户 / 最近回复”。
- 统一写入路径：用户消息入队与会话绑定、助手最终回复落地、图片回复落地。

### 阶段 2：统一展示与去 ID

- 统一标题与列表摘要选择逻辑。
- 移除 `/use`、切换提示、强踢确认、`/status` 中的默认短 ID 展示。

### 阶段 3：入口细化

- `/status` 增强为全量四元信息。
- 强踢确认卡增强为低频高风险决策视图（四元信息完整展示）。
- `/use` 保持信息密度可读（主信息 + 一条关键辅助信息）。

### 阶段 4：回归验证

- 覆盖 `/use`、会话切换、强踢确认、`/status`、request prompt。
- 验证旧线程（无历史摘要）与新线程（有完整摘要）均可读、无 ID 泄漏。
