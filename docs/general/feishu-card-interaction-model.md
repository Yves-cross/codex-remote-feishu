# 飞书卡片交互模型

> Type: `general`
> Updated: `2026-04-21`
> Summary: 固化当前产品讨论收敛出的飞书交互模型，明确单卡流程、确认流、共享过程卡、检查点卡与独立时间线项的边界。

## 1. 文档定位

这份文档回答的是产品层的问题：

- 飞书侧到底应该存在几种交互模型
- 哪些交互必须始终停留在同一张卡里
- turn 运行过程中，哪些内容属于持续 patch 的共享过程卡
- 哪些内容应该作为独立卡或独立消息插入时间线

它不负责描述具体 callback 字段、旧卡拒绝、daemon 生命周期 stamping 等实现细节。

这些内容分别以其他文档为准：

- 产品与业务卡片通用原则：
  - `docs/general/feishu-business-card-interaction-principles.md`
- 当前已实现的 Feishu UI 状态机：
  - `docs/general/feishu-card-ui-state-machine.md`
- 飞书卡片平台限制：
  - `docs/general/feishu-card-api-constraints.md`

## 2. 核心目标

这套模型的目标不是把所有内容都塞进一张大卡里，而是让用户始终清楚三件事：

1. 当前哪一张卡拥有这条交互
2. 当前看到的是流程卡、过程卡、检查点卡，还是普通时间线输出
3. 这张卡接下来会继续更新，还是已经封口

## 3. 术语

### 3.1 单卡流程

指一条从开始到结束都只认一张 owner card 的交互流程。

它可以在同一张卡内经历：

- 起始页
- 分步页
- 编辑页
- 执行中页
- 成功 / 失败 / 已取消终态页

但不应在流程中途逃逸成另一张新的主卡。

### 3.2 turn 时间线

指一次 turn 运行过程中，按时间向下追加的所有可见输出序列。

它不是单一一张卡，而是一条由多种可见项组成的时间线。

### 3.3 可见项

指最终真的渲染给用户的一条输出。

它可以是：

- 一张卡
- 一条文本
- 一张图片
- 一条 notice

如果某个内部事件最后没有被渲染出来，它就不算可见项。

### 3.4 打断

指当前活跃的共享过程卡被封口，不再继续 patch。

一旦发生打断，后续如果还有新的共享过程输出，必须新开一张共享过程卡。

## 4. 最终产品交互模型

当前应收敛为下面五类用户可见模型。

## 4.1 前台单卡流程

适用场景：

- 菜单进入后的业务向导
- slash command 打开的业务卡
- 工作区 / 会话选择
- 各类 picker 与其子步骤

产品要求：

- 整条流程只在同一张卡里完成
- 前进、返回、局部选择、校验失败、处理中、完成态都 patch 当前卡
- 不允许中途改成“原卡保留，结果另发主卡”
- 不允许插入“命令已提交”之类的过渡主卡

这类流程的典型判断是：

- 用户现在正在做一条明确的前台业务
- 用户注意力不应离开当前 owner card
- 当前卡应始终代表这条流程的真实最新状态

## 4.2 后台触发的确认流

适用场景：

- approval
- `request_user_input`
- MCP 授权确认
- 升级过程中的人工确认
- 其他后台主动跳到前台请求处理的确认动作

产品要求：

- 这类交互可以由后台主动发起
- 但从发起到结束，也只认这一张确认卡
- 确认、拒绝、取消、过期、完成都 patch 同一张卡
- 确认流结束前，不应再拆成多张新的主交互卡

它和前台单卡流程的差别只在触发源不同：

- 前台单卡流程是用户主动点进来
- 后台确认流是系统在过程中主动插进来

但底层语义上，它们都属于单卡流程。

## 4.3 共享过程卡

适用场景：

- 执行过程中的共享 progress
- tool / command / file change / compaction 等过程类输出
- 其他需要把最近一段过程聚合在同一张卡里持续 patch 的场景

产品要求：

- 任意时刻最多只能有一张活跃的共享过程卡
- 活跃共享过程卡永远是当前时间线里最下面的一张共享过程卡
- 这张卡只承载一段连续的、最终实际渲染出来的共享过程输出
- 同一段连续过程里，后续共享过程内容继续 patch 这张卡
- 一旦中间插入任何非共享过程的可见项，这张卡立即封口
- 后面若再有共享过程输出，必须重新开一张新的共享过程卡

共享过程卡不是整轮 turn 的唯一主卡。

它只是 turn 时间线里一段“最近过程窗口”。

## 4.4 检查点卡

适用场景：

- `plan` 卡
- `final` 卡
- 未来其他需要作为阶段里程碑保留在时间线中的卡

产品要求：

- 检查点卡是 append 型的独立卡
- 它不是共享过程卡的一部分
- 它出现时，会打断当前共享过程卡
- 它发出后保留在时间线上，随后可被后续消息逐渐往上挤

其中：

- `plan` 卡代表过程中的阶段性里程碑
- `final` 卡代表这轮 turn 的最终收口

`plan` 不应简单并入共享过程卡，因为它不是“最近过程窗口”，而是“应被保留下来的中途检查点”。

## 4.5 独立时间线项

适用场景：

- 正常的 agent message 文本
- 图片输出
- notice
- 其他最终独立渲染出来、但不属于共享过程卡或检查点卡的内容

产品要求：

- 这类内容按时间线独立追加
- 它们不是单卡流程的一部分
- 它们只要真的渲染出来，就会打断当前共享过程卡

## 5. 共享过程卡的严格规则

共享过程卡的关键不是“上游发来了什么事件”，而是“最终有没有渲染出别的可见项”。

因此它的分段依据必须按最终渲染结果来判断，而不能只按内部事件类型判断。

### 5.1 连续 patch 的条件

只有同时满足下面条件，当前共享过程卡才继续 patch：

1. 当前已有一张活跃共享过程卡
2. 新到达的内容最终会被渲染为共享过程内容
3. 从上一条共享过程内容到这次内容之间，没有插入任何其他非共享过程的可见项

### 5.2 打断条件

只要中间插入任意一条最终真的显示给用户的非共享过程可见项，当前共享过程卡就必须被打断并封口。

至少包括：

- 普通 agent message 文本
- 后台触发的确认流卡
- `plan` 卡
- `final` 卡
- 图片
- notice
- 其他独立卡或独立消息

### 5.3 不打断条件

如果某个中间事件因为产品可见性策略没有真正渲染出来，它就不应打断共享过程卡。

例如：

- 因 `verbose` 等级被压掉的过程事件
- 被聚合后没有单独显示的内部过程片段

这也是为什么共享过程卡必须以后端最终投影结果为准，而不能以前序事件流为准。

### 5.4 重开规则

当前共享过程卡被打断后，后面若再次出现可见的共享过程输出，则：

- 不能回头继续 patch 旧卡
- 必须新开一张新的共享过程卡
- 新卡继续成为当前时间线最下面、唯一活跃的共享过程卡

## 6. 典型时间线示例

### 6.1 纯过程到最终结束

```text
共享过程卡 P1 -> final 卡 F1
```

### 6.2 过程中插入普通文本

```text
共享过程卡 P1 -> 文本 T1 -> 共享过程卡 P2 -> final 卡 F1
```

### 6.3 过程中插入确认流

```text
共享过程卡 P1 -> 确认流卡 R1 -> 共享过程卡 P2 -> final 卡 F1
```

### 6.4 过程中插入 plan 卡

```text
共享过程卡 P1 -> plan 卡 L1 -> 共享过程卡 P2 -> final 卡 F1
```

### 6.5 多次交错

```text
共享过程卡 P1 -> 文本 T1 -> 共享过程卡 P2 -> 确认流卡 R1 -> 共享过程卡 P3 -> plan 卡 L1 -> 共享过程卡 P4 -> final 卡 F1
```

## 7. 当前功能到模型的映射

| 当前功能或输出 | 应归入的模型 |
| --- | --- |
| 菜单进入的业务向导、target picker、path picker、history 等 owner card 流程 | 前台单卡流程 |
| approval、`request_user_input`、MCP 授权、升级确认 | 后台触发的确认流 |
| exec/tool/file change/compaction 等最近过程窗口 | 共享过程卡 |
| `plan`、`final` | 检查点卡 |
| 普通文本、图片、notice | 独立时间线项 |
| reaction | 辅助微交互，不属于上面五种主模型 |

## 8. 收口方向

如果后续做结构统一，推荐按下面两层来收：

### 8.1 单卡流程引擎

统一承接：

- 前台单卡流程
- 后台触发的确认流

这两者共享的核心语义是：

- 始终只有一张 owner card
- 整轮交互在同一张卡里完成
- 结束前不拆出新的主交互卡

### 8.2 turn 时间线渲染层

统一承接：

- 共享过程卡
- 检查点卡
- 独立时间线项

这层的关键语义是：

- 共享过程卡只有一张活跃卡
- 检查点卡是 append 型里程碑
- 任何独立可见项都会打断共享过程卡

## 9. 不应误解的点

### 9.1 不是所有内容都应合进一张大卡

这套模型明确反对“一个 turn 永远只用一张卡承载所有内容”。

原因包括：

- `plan` 和 `final` 的语义不是滚动过程窗口
- 独立文本、图片、notice 本来就更适合作为时间线项存在
- 飞书卡片本身也有体积、组件数和更新频率限制

### 9.2 `plan` 不是共享过程卡的一部分

`plan` 会打断共享过程卡。

它更像过程中的里程碑卡，而不是共享过程卡里的一个小 section。

### 9.3 共享过程卡的分段依据是最终渲染结果

决定它是否被打断的，不是内部产生了什么事件，而是用户最终是否真的看到了别的可见项。

## 10. 后续使用方式

以后凡是设计或重构飞书交互时，先回答下面三个问题：

1. 这件事属于单卡流程、确认流、共享过程卡、检查点卡还是独立时间线项
2. 它是否会打断当前共享过程卡
3. 它完成之前，用户到底应该盯着哪一张卡

如果这三个问题答不清楚，就说明交互模型还没有定清楚，不应直接进入实现。
